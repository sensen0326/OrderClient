# 点餐小程序客户端需求文档

> 版本：v0.1  
> 更新时间：2025-12-10  
> 参考资料：`点餐小程序用户端需求文档.md`、云服务空间 `uniCloud-aliyun/database` 及 `cloudfunctions`

## 1. 背景与目标

- **项目背景**：当前客户端（C 端）仅提供静态菜单、简单购物车与占位页面（参考 `pages/index/index.vue`、`pages/menu/menu.vue` 等），尚未与后端运营配置、真实订单/支付/会员数据打通。云端仅初始化了 `zhuohao` 等对象，无法支撑全流程业务。
- **核心痛点**：缺少动态菜品管理、真实下单与支付链路、订单状态同步、桌台/排队能力、会员营销体系以及消息/售后触点（详见 F1-F25）。
- **建设目标**：
  1. 构建覆盖堂食与外卖的闭环点餐体验，打通“看菜单→加购→结算→支付→备餐/配送→评价/售后”的链路。
  2. 结合 `uniCloud` 后台（`uniCloud-aliyun/database` 与 `cloudfunctions`）实现菜品、订单、桌台、会员、营销、数据分析等能力的前后端统一。
  3. 输出可持续迭代的客户端体验规范，支持积分、优惠券、活动中心等增长玩法，并提供明确的验收口径。

## 2. 适用范围与用户角色

| 角色 | 主要需求 | 触达页面 |
| ---- | -------- | -------- |
| 堂食顾客（扫描桌码） | 查看实时菜单、多人拼单、免重复填写桌号、呼叫服务 | 首页、菜单、购物车、订单详情、消息中心 |
| 外卖顾客 | 选择地址/时间、校验配送范围、查看配送轨迹 | 外卖入口、地址簿、结算页、订单中心 |
| 会员用户 | 积分等级、优惠券领取与使用、积分商城兑换 | “我的”、会员中心、积分商城 |
| 售后/客服用户 | 申请退菜/退单、在线客服、查看处理进度 | 订单详情、客服页面 |
| 运营/门店人员（由后台管理） | 通过客户端实时获取桌台/排队状态、推送消息 | 消息中心、排队页面（显示状态） |

## 3. 业务流程（客户端视角）

1. **堂食扫码点餐**：顾客扫描 `restaurant_table` 的二维码 → 前端通过 `table.scanOpen` 绑定 `table_session` → 浏览 `dish_category`/`dish` → 多人拼单同步 `cart_sessions` → 结算 `checkout.preview` → 提交订单 `order.createFromCart` → 支付 `payment.prepare` → 状态推送/提醒。
2. **外卖下单**：顾客选择外卖 → 读取 `user_address`、`delivery_area` 判断配送范围 → 配置费用与优惠 → 支付后生成 `delivery_task` 并实时展示配送节点。
3. **排队/预约**：顾客在排队页面调用 `queue.take` 或 `reservation.create` → 接收 `message.center`/模板消息提醒 → 到店扫码入座后关联桌台。
4. **会员/营销**：登陆后前端获取 `user_profile`、`member_level_rule`、`point_ledger`、`coupon_template`/`coupon` → 展示等级、积分、优惠券，并在结算时调用 `coupon.applyCheck`、`point.consume`。
5. **售后与客服**：订单完成后触发评价入口 `review.create`，若出现问题则通过 `support.create` 建立工单；全程通知 `message.inbox`。

## 4. 功能需求

### 4.1 菜品与点餐（F1-F3）

1. **F1 菜品中心后台化**
   - 客户端需调用 `dishCategory.list`、`dish.list`、`dish.detail` 获取动态菜品、SKU、标签、推荐位，缓存 5 分钟并有失败兜底。
   - 支持门店维度 price/status，展示推荐角标、销量、评分。
   - **验收**：运营在后台调整 `dish.schema` 数据 5 分钟内在客户端生效；下架菜品在前端置灰不可加购。
2. **F2 规格/加料/库存管理**
   - 支持 SKU 价格/库存差异、限购、打包费展示；支持多口味/加料选项（来源 `option_groups` 字段）。
   - **验收**：同一菜品选择不同 SKU 后加入购物车价格正确；库存不足时即时提示。
3. **F3 菜品发现与搜索**
   - 首页/菜单页提供搜索框、热销/新品/推荐筛选、菜品评分展示、banner/推荐位（`dish_recommend_slot`）。
   - **验收**：关键词 2 秒内返回匹配菜品并定位分类；推荐位可在后台配置图片、跳转、排序。

### 4.2 购物车与结算（F4-F6）

1. **F4 多人拼单购物车**
   - 购物车数据写入 `cart_sessions`，以 `session_id` 关联桌码或外卖场景；支持参与人头像展示、实时同步、离线恢复。
   - **验收**：两台设备扫码同桌码共享购物车，网络恢复后自动合并。
2. **F5 结算信息完善**
   - 堂食结算自动带出桌号/人数/是否合单，外卖结算集成 `user_address`、送达时段、发票抬头、餐具份数等模板。
   - **验收**：堂食用户无须重复填写桌号；外卖用户可选择历史地址并完成配送范围校验。
3. **F6 优惠与费用拆解**
   - 结算页展示商品金额、打包费、配送费、服务费、可用优惠券/积分/满减；调用 `checkout.preview`、`coupon.applyCheck`、`point.consume` 计算抵扣。
   - **验收**：订单详情与支付页金额一致，异常时给出原因提示。

### 4.3 订单与支付（F7-F10）

1. **F7 订单创建与状态机**
   - 下单生成 `order`、`order_item`，完整记录费用、权益、状态日志；客户端展示状态进度条与节点时间。
   - **验收**：每笔订单存在唯一 `order_no`，状态与后台一致（待支付→备餐→完成等）。
2. **F8 微信支付/退款接入**
   - 集成 微信小程序支付，支持支付结果回调、补单、自动关单；前端展示支付中、成功、失败等场景。
   - **验收**：真实调起支付并在超时未支付后释放购物车/库存。
3. **F9 订单中心与售后操作**
   - `pages/order/order.vue` 替换为真实列表，支持按状态筛选、再来一单、取消/催单/申请售后、查看配送轨迹或呼叫服务。
   - **验收**：订单列表 10s 内加载完用户最近 6 个月记录，操作结果即时同步。
4. **F10 厨房/取餐通知**
   - 支持后厨叫号、语音提醒、消息推送（`kitchen_ticket`、`delivery_task`、`message.center`）。
   - **验收**：支付成功后订单自动推送到对应出餐口；状态变化触达用户。

### 4.4 桌台与外卖场景（F11-F13）

1. **F11 桌台管理与状态同步**
   - 扫码绑定 `restaurant_table` 状态，展示桌台占用/清桌等信息，可发起呼叫服务。
   - **验收**：后台变更桌台状态 1 分钟内前端实时更新；用户呼叫服务记录到 `table_session`。
2. **F12 排队与预约**
   - 前端提供取号、排队进度、叫号提醒、预约锁位功能，数据来源 `queue_ticket`、`reservation`。
   - **验收**：用户可实时查看排队状态并收到模板消息提醒，过号逻辑生效。
3. **F13 外卖地址与配送策略**
   - 地址簿读取/维护 `user_address`，并通过 `delivery_area`/`delivery_fee_rule` 计算配送策略，展示配送员信息、轨迹。
   - **验收**：超出范围提示不可配送，订单详情展示配送信息。

### 4.5 会员与营销（F14-F17）

1. **F14 登录与账号体系**
   - 支持微信一键登录、手机号授权、游客模式；入库 `user_profile`，可注销。
   - **验收**：首次进入“我的”页面需授权并写库，退出后可重新登录。
2. **F15 积分/等级体系**
   - 展示积分、等级进度、权益卡片，依据 `member_level_rule`、`point_rule`、`point_ledger` 实时计算。
   - **验收**：下单/签到后积分即时增加，达到等级阈值自动升级。
3. **F16 优惠券与活动中心**
   - 支持优惠券列表、领取、启用、使用规则校验；活动中心展示限时优惠、套餐、秒杀（`campaign`、`coupon_template`/`coupon`）。
   - **验收**：结算页自动提示可用券并校验叠加规则。
4. **F17 积分商城与权益兑换**
   - 展示 `point_goods`、`point_order`，支持虚拟券/实体商品兑换、物流/自提信息。
   - **验收**：积分兑换实时扣减库存并生成兑换订单。

### 4.6 售后与客服（F18-F20）

1. **F18 菜品与服务评价**
   - 支持图文评价、匿名、追评、敏感词校验；在菜品详情展示精选评价（`review`、`review_media`、`review_summary`）。
   - **验收**：每个已完成订单允许一次评价，1 分钟内后台可见。
2. **F19 客诉/客服通路**
   - 集成在线客服、电话、智能问答，支持退菜/退单、补偿等工单（`support_ticket`、`support_message`）。
   - **验收**：用户提交问题 5 分钟内看到处理状态更新。
3. **F20 消息与提醒**
   - 接入订阅消息、短信/企业微信推送及站内消息（`notification_subscription`、`message_center`），支持消息中心历史查看与退订。
   - **验收**：授权后可收到订单状态、排队叫号等提醒，并支持退订。

### 4.7 运营配置与数据（F21-F25）

1. **F21 运营组件可视化配置**
   - 首页、我的页面运营位（banner、九宫格、广告、底部文案等）由 `operation_slot`/`operation_config_version` 提供；客户端按版本拉取展示。
   - **验收**：运营发布后 1 分钟内客户端刷新生效，可灰度。
2. **F22 数据分析与埋点**
   - 客户端统一埋点（PV、下单转化、菜品热度、券使用、桌台翻台率）上报 `analytics_event`，支持查询近 30 天数据。
   - **验收**：关键页面/事件均有埋点，后台仪表盘可导出 CSV。
3. **F23 uniCloud 服务化接口**
   - 客户端所有业务调用 `serviceRouter`，内置鉴权、限流、错误码规范，支持分页与日志（参考 `cloudfunctions/common/uni-config-center` 等）。
   - **验收**：接口调用统一带 token/签名，错误码可定位。
4. **F24 运营/门店管理后台联动**
   - 客户端与 `uni-admin` 后台共享数据，支持角色与门店权限。客户端展示与后台保持一致。
5. **F25 运维监控与发布**
   - 虽属后端责任，但客户端需配合：提供版本号、埋点、错误上报（`cloud_log`），支持灰度发布、回滚提示。

## 6. 运行与非功能要求

- **性能**：核心页面（首页、菜单、订单列表）首屏加载 ≤2s，购物车同步延迟 ≤500ms，接口超时重试 3 次。
- **可靠性**：接口错误码统一（如 `C4001` 参数错误、`C5001` 系统异常），所有关键操作（下单、支付、退款）需幂等。
- **安全**：登录态使用 `uni-id` token，支付与个人信息需加密存储，重要接口开启频率限制。
- **可运维性**：客户端需输出版本号、设备信息至 `cloud_log`，配合灰度发布、崩溃上报；埋点需包含 `user_id`、`session_id`、`channel`、`table_no`。
- **可访问性**：重要信息（金额、桌号、状态）需文本展示，语音播报用于叫号提示。

## 7. 迭代里程碑与验收

1. **基础能力期（里程碑 M1）**：交付 F1-F8、F11、F14；完成堂食/外卖下单支付闭环。验收指标：真实支付成功率 ≥95%，订单状态同步 ≤5s。
2. **运营增长期（M2）**：交付 F9-F13、F15-F21；上线会员、营销、排队、配送能力。验收指标：会员登录率 ≥60%，排队叫号准确率 ≥98%。
3. **平台化期（M3）**：交付 F22-F25；完成运营后台联动、数据分析、运维流程。验收指标：运营组件变更 1 分钟内生效，关键埋点覆盖率 ≥95%。

## 8. 依赖与风险

- **依赖**：`uniCloud-aliyun/cloudfunctions`（订单、支付、配送、消息）、`uni-admin` 管理后台、微信支付商户号、第三方配送服务。
- **风险与对策**：
  - 云函数尚未全部实现 → 需同步后端迭代计划，先行 mock + 接口契约。
  - 多门店扩展 → 在客户端设计门店选择与路由能力，所有接口必须带 `restaurant_id`。
  - 消息订阅限制 → 提前提示用户授权并制定 fallback（站内信/短信）。

---

本需求文档覆盖客户端所需业务能力、数据映射与验收标准。后续若新增门店/场景，可在 F21-F25 的运营配置与数据体系内扩展，保持文档更新。
